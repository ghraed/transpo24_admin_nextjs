# Transpo_24 Admin

## Environment

Required frontend environment variables:

```env
NEXT_PUBLIC_API_URL=http://localhost:3001
NEXT_PUBLIC_SOCKET_URL=http://localhost:3001
NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=replace_with_your_google_maps_api_key
NEXT_PUBLIC_WEB_PUSH_VAPID_PUBLIC_KEY=
```

Only the public VAPID key belongs in Next.js. Never place the VAPID private key in this project.

## Browser Web Push

This admin app now supports browser Web Push for authenticated admin users.

- Service worker: `public/sw.js`
- Subscription endpoints used:
  - `POST /notifications/web-push/subscriptions`
  - `DELETE /notifications/web-push/subscriptions`
  - `GET /notifications/web-push/subscriptions/me` is supported by the backend but not required by the current UI flow

Behavior:

- The service worker is registered in the browser only.
- Notification permission is requested only after the user clicks the enable button in the dashboard.
- If permission is already `granted`, the app synchronizes the existing browser subscription after authenticated startup.
- During logout, the app removes the backend association and unsubscribes locally so the next admin on the same browser does not inherit the previous subscription.

Permission states:

- `default`: the app shows an enable action and does not open the browser permission dialog automatically
- `denied`: the app explains that permission must be re-enabled from browser site settings
- `granted`: the app reuses or creates the browser subscription and synchronizes it with the backend

## Local Testing

1. Set `NEXT_PUBLIC_WEB_PUSH_VAPID_PUBLIC_KEY` to the public key generated by the NestJS backend.
2. Start the backend and this admin app.
3. Open the dashboard on `localhost`.
4. Click `Enable notifications` from the dashboard card.
5. Confirm the browser permission prompt.
6. Trigger an admin Web Push event from the backend.

`localhost` is a secure context for development. Plain LAN HTTP addresses may not support all service-worker and browser notification features.

## Production Notes

- Production Web Push requires HTTPS.
- Browser Web Push coexists with the existing in-app toast notifications; it does not replace them.
- Service-worker updates follow the browser's normal update lifecycle.
- Different browsers may vary in how aggressively they surface background notifications or persist permission prompts.
